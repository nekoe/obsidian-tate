# DOM and UX Design Decisions

Created: 2026-04-15

## Vertical Writing via `contenteditable` div

`writing-mode: vertical-rl` is applied to a `contenteditable` div. Applying `writing-mode` to a `textarea` requires Chrome 119+, which may not be supported by the Electron version bundled with Obsidian. A `contenteditable` div is used instead for broader compatibility.

Text is read via `getValue()` (custom `serializeNode()` DOM walker) and written via `setValue()` (`replaceChildren(sanitizeHTMLToDom(parseToHtml(content)))`).

## Automatic DOM Event Cleanup

All `input` / `compositionend` / `paste` / `selectionchange` event listeners must be registered via `this.registerDomEvent(el, ...)` (direct `addEventListener` calls are prohibited). Using Obsidian's `Component.registerDomEvent` ensures listeners are automatically removed when `onClose` is called.

## File Switch Detection

The `file-open` workspace event is used (more reliable than `active-leaf-change`). `file-open` does not fire when the vertical writing view itself becomes active, so the currently displayed file is not accidentally reset.

## Plain-Text Paste

A `contenteditable` div prefers `text/html` from the clipboard by default, which would paste inline expansion span styles or external HTML styles. The `paste` event calls `e.preventDefault()`, then `e.clipboardData.getData('text/plain')` retrieves plain text only, which is inserted at the cursor position using the Selection/Range API.

```typescript
// Insert plain text at the current cursor position using Selection/Range API
const sel = window.getSelection();
if (!sel || sel.rangeCount === 0) return;
const range = sel.getRangeAt(0);
range.deleteContents();
range.insertNode(document.createTextNode(text));
range.collapse(false);
sel.removeAllRanges();
sel.addRange(range);
```

After paste, the `paste` handler in `view.ts` calls `commitToCm6()` immediately.

**Note**: `execCommand('insertText')` is deprecated and has been replaced with the Range API (Issue 9 fix).

## Auto-Indent

`text-indent: 1em` is applied via CSS (not saved to the file).

- Since `parseToHtml()` wraps each paragraph in a `<div>`, setting `text-indent: 1em` on `.tate-editor.tate-auto-indent` is sufficient — CSS inheritance applies it to each paragraph `<div>` automatically.
- While the user is typing new input (before any `<div>` has been created), `text-indent` on `.tate-editor` itself applies directly to the text, keeping indentation active at all times.
- `applySettings()` toggles the class with `el.toggleClass('tate-auto-indent', settings.autoIndent)`.

## Eliminating `innerHTML` (Issue 8 Fix)

All HTML insertion into the `contenteditable` div must use `sanitizeHTMLToDom()` + `replaceChildren()` / `insertBefore()`. Direct assignment to `innerHTML` is prohibited (flagged in the Obsidian Community Plugin review as Issue 8).

```typescript
// Before (prohibited):
this.el.innerHTML = parseToHtml(content);

// After (correct):
this.el.replaceChildren(sanitizeHTMLToDom(parseToHtml(content)));
```

`sanitizeHTMLToDom()` is an Obsidian-provided API that converts an HTML string to a `DocumentFragment`. It also serves as an XSS mitigation.
